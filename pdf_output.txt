Page 5 Step 2: How to Come Up with a Project Idea Many students feel stuck at this stage. Here is a structured four-step approach:  1. Start with a scenario, not a technology. Imagine a fictional organization: a small medical clinic, a law firm, a school district, a retail chain with an online store. What real-world security challenges would they face? 2. Pick a threat. What is the most realistic attack against that organization? Ransomware? Phishing leading to credential theft? An insider exfiltrating patient data? A compromised IoT device pivoting into the corporate network? 3. Design and build the defense. What combination of technologies, policies, and configurations would protect against that threat? This becomes your project. You must actually build the scenario (the organization’s environment), demonstrate how the threat impacts it, implement the defenses, and then show that those defenses are effective. 4. Map to modules. Check which course modules your defense touches. You need at least three. If you only have two, consider adding a monitoring, incident response, or governance component.   Project Requirement: Build, Attack, Defend Your project must include three phases: (1) Build the scenario — replicate the fictional organization’s environment using VMs and virtual networks. (2) Demonstrate the threat — show how the chosen attack actually impacts the environment before defenses are in place. (3) Implement and verify defenses — deploy your security controls and demonstrate their effectiveness through testing.   Brainstorming Prompt Complete this sentence: "A [type of organization] is vulnerable to [specific threat]. We will replicate this environment, demonstrate the attack, and then protect it by implementing [defense 1], [defense 2], and [defense 3]. We will verify effectiveness by [testing method]." That sentence becomes your mission statement.  Step 3: Define Your Mission Statement Based on your brainstorming, write a one-paragraph mission statement that clearly states: the type of organization, the security problem you are solving, how the threat will be demonstrated, and the key technologies you will use to defend against it. Page 6 Section 3: Understanding Agile Project Management on GitHub This section explains the Agile concepts you will use to manage your project. If you have never used GitHub or Agile before, read this section carefully before setting anything up. The Big Picture: How the Pieces Fit Together Think of your project as building a house. You would not try to do everything at once. You would plan phases (foundation first, then framing, then electrical, etc.) and within each phase you complete specific tasks. Agile project management works the same way. Here is how every concept connects:  Concept What It Is GitHub Feature Analogy Milestone A major project goal or deliverable phase Milestone (under Issues tab) "Finish the foundation" Sprint (Iteration) A 2-week time box of work Iteration (on the Project Board) "What we build this pay period" User Story A goal from a user’s perspective (the “why”) Issue labeled “user-story” (parent) "A room in the house" Task / Issue A specific piece of work (the “how”) Sub-Issue under a User Story "Install a window in that room" Story Points Effort/complexity estimate for a User Story Estimate field (parent issue) "How hard is that room to build" Project Board Visual board showing status of all work GitHub Projects (board view) "The whiteboard on the job site"  Milestones and Sprints: They Are Not the Same Thing This is a common source of confusion, so let us be precise. What Is a Milestone? A Milestone represents a major project goal or deliverable phase. It answers the question: "What do we need to have accomplished by this date?" A Milestone has a name, a description, and a due date. In GitHub, you create Milestones under the Issues tab of your repository. Examples of Milestones: • "Milestone 1: Project Setup and Planning" — Repository, wiki, architecture diagram, and proposal complete. • "Milestone 2: Base Infrastructure Deployment" — All VMs installed, network segments configured, basic connectivity verified. • "Milestone 3: Security Implementation and Attack Demonstration" — Defenses deployed, threat demonstrated, effectiveness tested.

Page 7 • "Milestone 4: Documentation and Presentation" — Final report, wiki, and presentation complete. What Is a Sprint (Iteration)? A Sprint (also called an Iteration on the GitHub Project Board) is a fixed-length work period — in our class, two weeks. During a sprint, your team commits to completing a set of tasks. At the end you review what was done and plan the next sprint. Sprints are configured as Iterations on the Project Board. Each Iteration has a start date and end date. How They Relate A Milestone can span multiple Sprints. For example, "Milestone 2: Base Infrastructure Deployment" might require work across Sprint 1 and Sprint 2 because the infrastructure is complex. Conversely, in a single Sprint you might complete the remaining tasks for one Milestone and begin tasks for the next.   Sprint 1 (Wk 2–3) Sprint 2 (Wk 4–5) Sprint 3 (Wk 6–7) Milestone 1: Setup Complete all tasks   Milestone 2: Infrastructure Begin VM setup Finish network config  Milestone 3: Security & Testing  Begin defenses Test & document  In GitHub: Two Separate Fields When you create an Issue in a repository that is linked to a GitHub Project, you will see two separate fields: • Milestone: Set in the Issue sidebar (repository-level). Assign the Issue to the Milestone it contributes to (e.g., "Milestone 2: Base Infrastructure Deployment"). • Iteration (Sprint): Set on the Project Board (project-level). Assign the Issue to the Sprint during which you plan to work on it (e.g., "Sprint 1").   Key Rule Every Issue must have both a Milestone and an Iteration assigned. The Milestone tells you WHAT goal the work contributes to. The Iteration tells you WHEN you plan to do it. An Issue without both will be difficult to track.  How to Define Good Milestones Each Milestone should have a clear, measurable goal. Ask: "How will we know this Milestone is done?" If you cannot answer that, the goal is too vague. • Good: "All VMs are deployed, network segments are configured, and connectivity between zones is verified with screenshots." • Vague: "Work on infrastructure." (What specifically? How do you know when it is done?)

Page 8 Why Does GitHub Call Everything an “Issue”? In GitHub, the word "Issue" is the name of the platform’s built-in tracking feature. It is used for everything: bugs, tasks, feature requests, questions, and user stories. The name comes from software development, where teams originally used Issues to track bugs and problems — but over time the feature expanded to cover all types of work. In this class, you will use Issues in two ways: • User Story Issues (Parent Issues): These represent high-level goals. Label them "user-story" so you can tell them apart from tasks. Assign story points (Estimate) to these. • Task Issues (Sub-Issues): These represent the specific steps to complete a User Story. Create them as sub-issues under the parent. Assign Size (XS–XL) to these.  The terms "task," "issue," and "sub-issue" are often used interchangeably in conversation. Just remember: in GitHub, the feature is always called an Issue, regardless of what it represents. User Stories and Tasks: The Full Breakdown User Stories: The “Why” A User Story describes a goal from the perspective of someone who benefits from the system. It follows this template:  "As a [role], I want to [action] so that [benefit]."  For a security project, examples include: • "As a network administrator, I want to segment the network into VLANs so that a compromise in one zone does not spread to others." • "As a security analyst, I want to deploy a SIEM and configure custom alert rules so that I can detect brute-force login attempts across all monitored systems." • "As a penetration tester, I want to demonstrate a phishing attack against the test environment so that we can measure whether our email filtering and user training controls are effective."  In GitHub, create a User Story as an Issue, label it "user-story," and assign it an Estimate (story points). This Issue becomes a parent that holds sub-issues (tasks). Tasks (Sub-Issues): The “How” Tasks are the specific, actionable steps needed to complete a User Story. In GitHub, create them as Sub-Issues under the parent User Story. Page 10 Story Points: Where Do They Go?   The Rule for This Class Assign story points (the Estimate field) ONLY to User Stories (parent issues). Do NOT assign story points to individual tasks (sub-issues). Instead, use the Size field (XS, S, M, L, XL) on tasks to indicate relative effort.  Why This Approach? Story points measure the total effort, complexity, and uncertainty of delivering a complete goal (the user story), not the effort of each individual step. Think of it this way: if you are estimating how hard it is to build a bathroom, you give one estimate for the whole bathroom (say, 5 points). You do not score each fixture separately and add them up. The story points capture the overall picture, including risks and unknowns. How to Assign Points: A Practical Guide Use the following scale. When deciding between two values, ask: "How much uncertainty is there? Have we done something like this before?"  Pts Complexity Criteria Example User Story 1 Trivial Exactly one step, no uncertainty, takes under 30 minutes "Update README with team names and project title" 2 Small 1–2 straightforward steps, you have done this before or a clear tutorial exists "Install a single VM from a known image and verify network connectivity" 3 Medium 3–4 steps, some configuration research required, low-to-moderate uncertainty "Configure pfSense with NAT rules and verify LAN→WAN traffic" 5 Large 5+ steps, involves integrating multiple components, moderate uncertainty, may require troubleshooting "Deploy Wazuh SIEM, install agents on two endpoints, and create a custom alert rule for failed SSH logins" 8 Very Large Many steps, high uncertainty, dependencies on other stories, team has no prior experience with the technology "Build a two-tier PKI with offline Root CA, online Sub CA, issue certs to a web server, and verify client trust chain"  When Should You Use 8 Points? An 8-point story should be rare. It signals that the User Story is at the upper boundary of what your team should attempt in a single story. Before assigning 8 points, ask these questions: 1. Can this story be split into two smaller stories that each deliver independent value? If yes, split it. 2. Does the team have zero prior experience with the core technology? If yes, 8 points may be justified because of the learning curve.

For tasks (sub-issues), use the Size field on the Project Board instead of points: • XS = Less than 30 minutes of work • S = About 1 hour • M = A few hours • L = A full work session (half a day) • XL = Too big. Break it into smaller tasks.

Key Terminology • Repository (Repo): A project folder on GitHub containing your files, history, and settings. • Issue: GitHub’s built-in tracking item. Used for user stories, tasks, bugs, and questions. Think of it as a task card. • Sub-Issue: An Issue nested under a parent Issue. Used to break User Stories into Tasks. • Milestone: A goal container that groups Issues by deliverable phase, with a due date. • Iteration (Sprint): A time-boxed period on the Project Board. Configured in the Project settings. • Project Board: A Kanban-style board (Backlog → In Progress → Done) that visualizes your Issues. • Wiki: Built-in documentation pages attached to your repository. Your final report lives here. • Label: A color-coded tag for categorizing Issues (e.g., “user-story,” “bug,” “documentation”).  The Workflow in 6 Steps 1. Create Milestones (one per major project phase) with due dates and deliverable descriptions. 2. Configure Iterations (Sprints) on the Project Board with two-week durations. 3. Create User Story Issues (labeled “user-story”), assign Estimate (points) and a Milestone. 4. Create Task Sub-Issues under each User Story, assign team members, Size (XS–XL), and the same Milestone. 5. Set the Iteration (Sprint) field for each Issue on the Project Board. 6. Move cards across the board as work progresses: Backlog → Ready → In Progress → In Review → Done.

Page 22 Appendix C: Frequently Asked Questions  Q: Do I assign story points to tasks or to user stories? Assign story points (Estimate) only to User Stories (parent issues). Tasks get a Size (XS–XL) instead. Story points measure the overall effort of delivering a complete goal, not individual steps. Q: What if a task turns out to be bigger than expected? If a task is too large, break it into smaller sub-issues. If a User Story is too large (8+ points), consider splitting it into two separate User Stories. Q: What is the difference between a Milestone and an Iteration (Sprint)? A Milestone is a project goal with a due date (e.g., "Infrastructure Deployed"). An Iteration/Sprint is a two-week time box during which you work on tasks. A Milestone can span multiple Sprints. When you create an Issue, you assign it to a Milestone (what goal it contributes to) and an Iteration (when you plan to do it). Q: How many story points should we plan per sprint? For a 2–3 person team in a two-week sprint, plan 8–15 points total. Start conservatively in Sprint 1 (around 7–10 points). As you learn your team’s pace, adjust. Q: Can I change the sprint plan mid-sprint? Yes. If you finish early, pull tasks from the Backlog. If something takes longer than expected, move incomplete items to the next Sprint’s Iteration. Document the change in your Wiki sprint notes. Q: Why are tasks, issues, and sub-issues all called different things? In GitHub, everything is an "Issue" — that is the name of the platform feature. "Task" is the Agile term for a specific piece of work. "Sub-issue" describes the parent-child relationship. They refer to the same thing viewed from different angles: GitHub calls it an Issue, Agile calls it a Task, and when it is nested under another Issue we call it a Sub-Issue. Q: Does every team member need to do technical work? Yes. Every team member must contribute to building, configuring, and testing the systems. Roles like "Project Manager" or "Documentation Lead" define who coordinates those areas, but every member must also be hands-on with the technical implementation. Q: How do I pick a project idea? Consider three factors: your team’s interest (you will spend weeks on this), the resources available (VMs, cloud credits), and the module coverage (minimum three). Use the brainstorming framework in Section 2 to generate custom ideas if the examples in Appendix A do not fit.
